# 审核框架（Audit Framework）

> **定位：** 本模块是 deep-intent-collaboration 「审核框架」横切段的完整方法论展开。
> 当 AI 需要做方案审核（选前）、产出审核（交付前）、自洽审核（全程）或逼完整压测（采信关键结论前）时由主技能引用加载。
> 横切贯穿全程——不属任何单一阶段，是覆盖协作完整生命周期的校验机制。

## 顶层使命

**审核不是阶段二的子步骤，而是贯穿全流程的横切机制。**
三种审核类型覆盖协作的完整生命周期——方案审核（选前）、产出审核（交付前）、自洽审核（全程）。
共享逻辑学手段作为校验引擎，不归属于任何一个审核类型。

每个动作都能反查元依据：**"这个判断/产出，经得起逻辑校验吗？"**

## 三种审核类型总览

| 审核类型 | 时机 | 对象 | 核心手段 |
|---------|------|------|----------|
| **方案审核** | 阶段 1.5 → 阶段二切换时 | 多个候选方案的对比选择 | 6 维对比 + 三段论 + 反证法 + 假设条件分析 |
| **产出审核** | 阶段二交付前 | 单一交付物的质量 | 自顶向下分层 + 整体-部分关系 + 概念界定 + **论证质检五步（结构层）+ 事实查证三层（事实层）** |
| **自洽审核** | 全程（任何判断产生时） | 选择 vs 锚点 | 假言推理 + 否定后件 + 关系自洽 + 因果 5 问 |
| **逼完整引擎** | 生成中（采信关键结论前） | AI 即将输出的论证 | **四压测：多立场 / 最强反方（呈现）/ 边界 / 模拟反驳（自检）** |

**审核引擎的两侧：** 前三种审核类型都是"挑漏洞"侧（问"对不对"），逼完整引擎是"逼完整"侧（问"全不全"）。论证滴水不漏（对），仍可能没回答意图（不全）——两侧互补，不是重复。

```
                 ┌──────────┐
                 │ 自洽审核  │ ← 全程兜底：所有选择 vs 锚点
                 │ (全程)    │
                 └────┬─────┘
                      │ 自洽通过后才进入
          ┌───────────┴───────────┐
          ▼                       ▼
    ┌──────────┐           ┌──────────┐
    │ 方案审核  │           │ 产出审核  │
    │ (选前)    │           │ (交付前)  │
    └──────────┘           └──────────┘
    确保选的对              确保做的好

         ┌──────────────────────────────────────┐
         │ 以上三种都是"挑漏洞"侧（问"对不对"）  │
         │                                      │
         │ 逼完整引擎（四压测）是"逼完整"侧       │
         │ 生成中并行运行，采信关键结论前至少一项 │
         │ 确保想的全（问"全不全"）              │
         └──────────────────────────────────────┘
```

## 横切要求·逻辑校验可视化（ASCII 关系图，优先树状）

**核心命题：逻辑校验的对象是"关系"——前提与结论、锚点与选择、论点与论据——关系天然是结构。用线性文字叙述 = 把结构压扁，校验者和被校验者都看不清哪根线断了。**

**凡使用本框架任一逻辑手段（9 大手段 + 四压测）执行校验，必须用 ASCII 图把被校验的关系画出来，优先树状图。** 不画 = 关系隐在线性文字里 = 断链 / 隐藏前提 / 矛盾线全都看不见 = 校验沦为"我觉得自洽"。

**为什么是图不是文字（每种手段压成文字后丢的东西）：**

| 手段 | 被校验的关系 | 压成文字丢什么 |
|------|------------|--------------|
| ①三段论 | 大前提-小前提-结论（三层） | "隐藏前提"藏在哪看不出来 |
| ②反证法 | 锚点 vs 非A 的推导链 | 矛盾出在哪一步被叙述稀释 |
| ③假设条件 | 方案↔多个前提条件 | 哪个条件 ❌/⚠️ 不直观 |
| ④整体-部分 | 骨架-模块-细节的支撑链 | 哪层断了要逐句找 |
| ⑥论证质检 | 前提-隐藏前提-结论（分叉） | "哪根前提断了"完全不可见 |
| ⑧⑨自洽审核 | 锚点-选择-矛盾（网状） | "矛盾线"被压成普通句子 |
| 四压测 | 正方-反方-折中（并行分支） | "全貌"压不出，反方被弱化 |

**作图三要求：**

1. **优先树状图**——关系是层级/从属/分叉的（前提→结论、锚点→选择→子选择、论点→论据→子论据），用树。树把"谁支撑谁、谁反驳谁"一层层摊开，断链一眼可见。
2. **关系图（非层级关系时）**——关系是互斥/循环/网状的（约束互斥、循环论证、多对多），用节点+连线，标注关系类型（支撑/约束/互斥/反驳）。
3. **标注校验结果**——图上每个节点/连线标校验状态：`✅ 成立` / `❌ 断裂` / `⚠️ 待验` / `🤖 AI 可判` / `👤 用户定`。校验结果落在结构上，不是另起一段文字复述。

**示例（手段① 三段论校验，树状）：**

```
选择方案 A（结论）
├─ 大前提：本质需求是 X（锚点）✅ 阶段一已校验
├─ 小前提：方案 A 满足 X ⚠️ 待验
│   ├─ 支撑：[依据1] 🤖
│   └─ 隐藏前提：[满足X但不违反Y] ❓ 未审 → 断链风险
└─ 校验结果：小前提待验 + 隐藏前提未审 → 不自洽，回补后再定
```

**示例（自洽审核·关系图，锚点-选择有张力）：**

```
[本质需求锚点] ──支撑──→ [选择 A]
       │
       └──互斥 ⚠️──→ [约束 B：与 A 冲突]

矛盾线：锚点既要 A 又受 B 约束，A 与 B 互斥 ❌ → 不自洽，显式摆给用户 👤
```

**适用范围：** 本框架全部 9 大手段（①三段论 ②反证法 ③假设条件 ④整体-部分 ⑤概念界定 ⑥论证质检五步 ⑦事实查证 ⑧假言推理 ⑨否定后件）+ 四压测（多立场 / 最强反方 / 边界 / 模拟反驳）——**凡执行校验即作图**，无一例外。

> 本要求是主技能「输出排版原则」（关键信息不可视觉弱化）+「决策透明化」在逻辑校验场景的落地——校验过程本身是关键判断，埋进线性文字 = 视觉弱化 = 校验做了等于没做。与 `external-solution-research.md`「论证结构可视化」、`learning-design-flow.md`「贯穿要求·论证结构可视化」同源：三者都是"把关系从线性文字还原成可见结构"，本节管校验时，那两节管产出时。

## 审核类型一：方案审核（选前）

**触发时机：** 阶段 1.5 结束、用户准备做方案选择时。

**输入：** 阶段 1.5 的 6 维评估矩阵 + 差异分析 + 影响分析

**输出：** 逻辑校验后的方案推荐 + 用户确认的选择

### 手段①：三段论校验

将方案选择转化为三段论，检验逻辑链条是否断裂：

```
大前提（锚点）：本质需求是 X
小前提（方案）：方案 A 满足 X
结论（选择）：选择方案 A 是合理的

校验问题：
- 大前提是否成立？→ 锚点是否通过了阶段一的逻辑自洽校验？
- 小前提是否成立？→ "方案 A 满足 X" 是事实判断还是假设？
- 结论是否自洽？→ 有没有隐藏的前提被忽略？（"满足X但违反Y"）
```

**话术模板：**
> "我从逻辑上拆一下这个选择：
> - 你的本质需求是 [锚点]（大前提）
> - 方案 A 满足这个需求（小前提）
> - 所以选 A 是合理的（结论）
>
> 校验：小前提的『满足』有依据吗？有没有『满足X但违反Y』的隐藏前提？"

### 手段②：反证法

假设选了相反方向，推导是否导致矛盾：

```
假设选「非A」（或选B、C）→ 推导 →
  → 是否与锚点矛盾？→ 反证 A 是必要选项
  → 是否与约束矛盾？→ 反证该方案不可行
  → 是否比 A 更好？→ 反证 A 不是最优
```

**话术模板：**
> "做一下反证：假设我们选 B 而不是 A——
> B 是否和你的 [某约束] 冲突？如果有冲突，说明在这个维度上 A 几乎是唯一可行选择了。"

### 手段③：假设条件分析

把方案成立的前提条件显式化，检验这些条件是否满足：

```
"方案 A 成立需要：
条件1：数据库必须支持事务 → ✅ Postgres 支持
条件2：团队有 Rust 经验 → ❌ 主栈是 TypeScript
条件3：上线前需要 QA 周期 → ⚠️ 由用户判断

条件1-2 是事实判断，AI 可评估。
条件3 是用户判断，交给用户。
不把用户判断项替 AI 判了。"
```

## 审核类型二：产出审核（交付前）

**触发时机：** 阶段二每步交付物产出后、交给用户确认前。

**输入：** 阶段二的交付物（代码、文档、方案、报告……）

**输出：** 经自顶向下校验的交付物 + 用户确认

**实现：** 即主技能阶段二「产出审核披露铁律」（拉回对话框 + 自顶向下逐层审核）。此处补充逻辑学手段。

### 手段④：整体—部分关系校验

```
检验交付物的部分与整体是否自洽：

1. 整体结构是否覆盖了阶段一确认的所有约束要素？
   → 有遗漏 = 结构有缺口

2. 各部分之间是否有逻辑依赖关系？
   → 有依赖 = 被依赖的部分是否有对应内容？

3. 细节层是否支撑模块层，模块层是否支撑骨架层？
   → 自顶向下 + 反向校验：从细节反推到骨架，链条是否断裂？
```

### 手段⑤：概念界定校验

```
检查交付物中是否使用了抽象/模糊概念：

"系统应具有良好的性能" → ❌ "良好"没有操作化定义
"系统的 API 响应时间 < 200ms" → ✅ 可测量可验证

原理：未界定的概念不能作为推理的前提。
阶段一要求约束/成功标准可观察可验证——产出审核同样适用。
```

### 手段⑥：论证质检五步（结构层 · 拆→补→判型→谬误→移交）

> **目标：对交付物里的论证（前提→结论）做系统质检。**
> 逻辑学只审"论证是否成立"，不审"前提是否为真"——后者由事实层（手段⑦）接住，各管一摊。

**五步顺序不可跳：**

**步骤 1：拆** —— 分清命题 vs 论证
- **命题**（陈述句，真假靠查证）：单独陈述，不展开推理。
- **论证**（前提→结论）：逻辑学地盘。
- 画三块：**前提？/ 结论？/ 推理那步？** 混在一起就拆不清。

**步骤 2：补** —— 挖出省略的隐藏前提（最常被忽略，最关键）
- AI 给的几乎都是**省略三段论**，藏的最没人查。
- 三步法：定位结论 → 找已给前提 → 问"**凭什么？**"补出隐藏前提，重点审。
- 反例：AI 说"方案 A 可行" → 隐藏前提可能是"团队有相关经验"——这个前提未必成立。

**步骤 2 结构性闸门（防"走形式"糊弄）：**
步骤2处理的是"没说出口"的东西，产出最难外部验证，最易被偷懒跳过。因此设硬要求：
**每一条已给前提，都必须各自补出对应的隐藏前提，并逐条判断其真假——不止"重点审"。**
- ✅ 合规："已给前提 P1 → 隐藏前提是 Q1，Q1 成立/不成立；P2 → Q2，..."
- ❌ 走形式："已给前提有一些隐藏前提，重点审过了"（没逐条展开 = 等于没做）
- **步骤2的价值，恰恰是挖出那些不属于标准谬误标签、但最致命的隐藏假设**（如"收益>成本从未被审视"）——这类前提在步骤4的谬误清单里反而不显眼，跳过步骤2就会漏掉。

**步骤 3：判型** —— 区分三种推理，各查各的
| 推理类型 | 查什么 | 常见陷阱 |
|----------|--------|----------|
| **演绎** | 有效性（形式对不对）+ 可靠性（前提真不真） | 形式对但前提假 = 无效结论 |
| **归纳** | 样本代表性（结论只是概率，不是必然） | AI 最爱把"几次经验"当普遍规律 |
| **溯因** | 有没有被忽略的更好解释 | AI 最爱给"唯一合理解释"——必问"**还有别的可能吗？**" |

**步骤 4：查谬误** —— 对 7 大谬误过一遍 + 盯量词升降
- **7 大谬误**：①乞题（循环论证）②混淆必要/充分 ③以偏概全 ④相关当因果 ⑤虚假两难 ⑥歧义（偷换概念）⑦诉诸权威
- **量词升降**：读到"有些/某些/有时"，立刻查后面有没有被偷偷改成"大多数/总是/普遍"。**从存在推全称是非法跳跃**——这是 AI 输出里最高频的逻辑错误。

**步骤 5：移交** —— 结构没问题 ≠ 结论成立
- 论证形式正确、链条不断，只能说明"如果前提真则结论真"。
- 前提是否真？交给事实层（手段⑦）查证。

### 手段⑦：事实查证三层（事实层 · 来源/可信度/可证方法）

> **目标：对交付物里无法直接判断真假的论据，逼出可验证性。**
> 与决策透明化框架（`decision-transparency.md`）的「信息溯源🔗」和「确信度🟢🟡🔴」配合——溯源是标出来源，这里是审核来源够不够硬。

**每个存疑论据，要求 AI 给三样之一：**

| 要求 | 逼什么 | 为什么 |
|------|--------|--------|
| **可追溯来源** | 论文/官方文档/原始数据/链接 | 逼它给具体出处才能查 |
| **可信度** | 权威性、时效性、原始还是二手 | 有来源 ≠ 可信 |
| **可证方法** | 给不出处时，说清怎么验证 | 说不清怎么验证就不是事实 |

**硬规则：查无可靠来源、又说不清怎么验证的，一律当未证实，不采信。**
**越是精确具体的数据（"73% 的用户……"），越要查**——精确的数字最容易被编造，也最有迷惑性。

**来源无关性铁律（防"用户传来的数字"绕过查证）：**
数字是否需要查证，**只看它有没有可追溯来源，不看它从谁的嘴里说出来**。AI 编的数字要查，用户带进来的数字同样要查——**记忆 ≠ 来源，"我之前在别处见过"不构成出处**。
- 用户说"这个 65% 我之前也见过" → 仍按未证实处理，反问"在哪见过？能把原始链接/报告发我吗？"
- 发得出来 → 一起核；发不出来 → 哪怕用户很确定见过，依然停在这一档，不放行
- **为什么如此严格**：二手传播（用户回忆、博客转述、群里看到）正是这类查无出处数字的主要扩散路径。规范若只防 AI 编、不防用户传，等于防了上游不防下游——数字照样混进来。
- **身份不变**：无论数字来自 AI 还是用户，过的是同一个三层。用户身份不降低查证门槛，恰恰相反——用户带进来时 AI 更容易因"用户说的应该可靠"放松，这正是要防的拉力。

**话术示例：**
> "你引用的 [某数据]，我查证一下：
> - 可追溯来源？→ [是/否，具体出处]
> - 可信度？→ [权威性/时效性评估]
> - 可证方法？→ [给不出处时怎么验证]
>
> 三个都拿不出来，这条按未证实处理，不算入结论。"

**话术示例（用户带进来的数字）：**
> "你说这个数据'别处见过'——**见过不等于有出处，记忆不是来源**。能把当时看到的原始链接/报告发我吗？发得出来我们一起核；发不出来，它和 AI 凭空编的没区别，依然按未证实处理。"

## 审核类型三：自洽审核（全程）

**触发时机：** 任何选择/判断产出时（不仅限于阶段一锚点固化）。

**输入：** 当前选择 vs 锚点

**输出：** 自洽通过（继续）或不自洽（矛盾显式摆给用户）

**与主技能的衔接：** 主技能阶段一「逻辑自洽校验」（锚点固化时的特例）+ 阶段二「分叉点校验步骤 3」（分叉点的特例）都是本审核类型的特例应用。此处将逻辑抽象为通用手段。

### 手段⑧：假言推理（If-Then 校验）

```
"如果做了 A，那么会得到结果 B"
→ 校验 1：A → B 的因果关系是否成立？
→ 校验 2：有没有第三变量 C 同时导致了 A 和 B？（相关 ≠ 因果）
→ 校验 3：B 是否真的对齐用户锚点？

这是阶段一「因果自洽 5 问」的通用化——从锚点校验扩展到任何判断。
```

### 手段⑨：否定后件（Modus Tollens）

```
"如果锚点成立，那么选择 A 应该合理"
→ 观察：选择 A 导致了矛盾后果
→ 结论：要么锚点不自洽，要么 A 实际不满足锚点
→ 动作：回到锚点或方案重新校验
```

### 自洽审核的触发时机速查

| 触发点 | 校验什么 | 用什么手段 |
|--------|----------|-----------|
| 用户在阶段 1.5 做选择时 | 所选方案 vs 锚点 | 三段论 + 反证法 |
| AI 在阶段二推进一步后 | 这一步的产出 vs 锚点 | 假言推理 + 整体-部分关系 |

## 审核引擎的两侧：逼完整 + 挑漏洞

**核心命题：逻辑工具两面用——事前逼 AI 主动补全（问"全不全"），事后逼 AI 露破绽（问"对不对"）。**
上面的三种审核类型（方案/产出/自洽）都是"挑漏洞"侧：判断产出后挑毛病，确保"对"。
但论证滴水不漏（对），仍可能没回答意图（不全）——"看着完整却藏未检验前提"是另一种失败。
逼完整引擎补这一侧，与挑漏洞互补，不是重复。

### 逼完整引擎：四压测（生成中，采信关键结论前至少做一项）

**触发：** AI 准备采信/输出**关键结论**前（不是审核时机，而是生成时主动做）。

**"关键结论"的判定（不可自定义绕过）：** 凡满足以下任一即为关键结论，必须压测——
- **影响用户选择**（推荐 A 而非 B、倾向某方案、建议采取某行动）
- **不可逆操作**（删除、提交、发布、外发、生产环境改动）
- **作为后续推理前提**（其他判断建立在它之上）
- **用户未明确要求但 AI 主动给出的判断**（不是简单复述事实）

**红线：** AI 不得把本应压测的推荐自我标定为"常识陈述/简单事实"绕过触发。判断"这句话算不算关键结论"时，宁可压测（压错了只是多花精力），不可放过（放过了可能漏未检验前提）。

| 压测 | 让 AI 做 | 杀什么 | 性质 |
|------|----------|--------|------|
| **多立场论证** | 把各方论证写一遍再下结论 | 隐藏的反方 | 呈现给用户（高风险节点） |
| **最强反方** | 给出反对结论的最强理由 | 自我证实偏差 | 呈现给用户（高风险节点） |
| **边界压测** | 极端条件下结论是否还成立 | 看似成立实则崩塌 | AI 内部自检（全程） |
| **模拟反驳** | 扮演对手逐环反驳 | 论证链条薄弱处 | AI 内部自检（全程） |

**分工与呈现策略（压测必做，呈现可选）：**

四压测的"做"和"给用户看"是两件事——**做是硬步骤（质量保证），看是可选（认知负荷管理）**。

**第一层：做不做 → 必做 + 留痕。** 采信关键结论前，四压测至少做一项，这是逼完整引擎的硬要求，不可因"用户跳过"省略。省了压测 = 论证未经"逼完整"检验 = 可能"看着完整藏未检验前提"。

**留痕铁律（防"默默内化"变成"根本没做"）：** "默默内化"不是黑箱——即使结果不展开给用户，**压测的结论也必须以一句话背书的形式落到输出里**：
- 高风险节点："我做过压测（最强反方是 X），结论是 Y"——呈现后用户看得见
- 常规节点："（已压测）倾向 Y"——括号背书是留痕，证明压测发生过

**只有括号背书、没有实质压测内容的，等于没做。** 括号背书不是橡皮图章，它要求背后真有压测动作。若被质疑"压了什么"，AI 必须能展开具体的反方/边界测试内容——展开不出 = 造假。

**第二层：给谁看 → 分项归属。**

| 压测 | 归谁 | 理由 |
|------|------|------|
| **多立场论证** | 👤 呈现给用户审 | 让用户看清全貌，藏起来 = 替用户过滤信息 = 越界 |
| **最强反方** | 👤 呈现给用户审 | 让用户看见反对意见，是"增强人"的核心价值 |
| **边界压测** | 🤖 AI 内部自检 | 纯技术性（极端条件崩没崩），用户看了无可判断的决策点 |
| **模拟反驳** | 🤖 AI 内部自检 | 纯技术性（链条哪环弱），同上 |

**第三层：何时呈现 → 只在高风险/不可逆节点询问，常规默默内化。**

多立场和最强反方吃认知载荷（要消化 3 套论证 + 理解 AI 回应再判断），每次推荐前都问 = 打扰。按节点风险分级：

| 节点性质 | 呈现策略 |
|----------|----------|
| **高风险/不可逆**（外发、删除、提交、发布、不可撤回的选型） | **呈现前询问**："我做了压测，要不要看多立场+最强反方？" 让用户选进入或跳过 |
| **常规推进**（可逆、低风险、快速对齐） | **默默内化**：压测结果并入推荐，不展开、不询问。只说"我做过压测了"作为推荐的可信度背书 |

**跳过的两种含义（用户选"跳过"时）：**
- 用户跳过"呈现" ✅ 允许——压测结果仍内化进推荐，AI 用了用户没看见。质量未让渡，只让渡了信息可见性。
- 用户跳过"压测本身" ❌ 禁止——逼完整引擎是硬步骤。"用户说跳过"不能成为 AI 省略自检的退路。

**圆外砍刀：** 压测只对"圆内论证"做（落在意图/边界圈内的部分）。圈外的论证直接砍，不必压测——避免对无关内容浪费精力。

**话术模板（高风险节点，呈现前询问）：**
> "这里是个不可逆选择，我倾向 [结论]。推荐前我做了压测——
> 要不要看**多立场论证**和**最强反方**？看的话你能基于完整信息判断，不看的话我直接给推荐（压测结果仍已内化）。"

**话术模板（用户选看时，展开呈现）：**
> "在我推荐 [结论] 前，做了两步压测：
> - **多立场**：支持方论证是……；反方主张……；折中派论证是……
> - **最强反方**：反对它的最强理由是 [X]。我的回应是 [Y]，但 [X] 是否成立你来判断 👤
>
> 基于这两步，我倾向 [结论]，但你需要看反方意见后定夺。"

**话术模板（常规节点，默默内化）：**
> "我倾向 [结论]（已做过压测，论证经检验）……"

## 审核闭环：不过怎么打回

> 审核不是"过了就交付，不过就全盘重来"。按**问题类型分流打回**，不一律退回原点。

```
审核结论
  │
  ├── 圆内自洽 + 三层全过 ──→ 交付
  │
  ├── 价值层不过（偏离意图）──→ 回意图层重定标尺   [方向问题]
  │
  └── 结构层/事实层不过      ──→ 回生成层让 AI 修正  [质量问题]
```

| 审核结果 | 打回到 | 原因 |
|----------|--------|------|
| **价值层不过**（方案/产出偏离本质需求锚点） | 阶段一意图层，重定标尺 | 问题在方向，改产出没用，得重新对齐意图 |
| **结构层/事实层不过**（论证链断裂 / 前提不可信 / 谬误） | 当前生成层，让 AI 针对性修正 | 问题在质量，方向没错，重做意图是浪费 |

**原则：方向错 → 退得远（回意图）；质量差 → 退得近（回生成）。不混淆两类问题，避免"方向对了却反复重定意图"或"方向错了却只在产出层打补丁"。**

**判别标准（防误分类洗白）：**
分流的有效性全押在"正确识别是哪一层不过"上。价值层和结构层有灰区，可互相洗白（把质量硬说成方向→全盘重来；或把方向硬说成质量→只在产出层补丁）。以下硬判别线堵这个逃逸通道：

| 判别问题 | 价值层（方向问题） | 结构层/事实层（质量问题） |
|----------|-------------------|------------------------|
| **替换测试**：把论证修复到完美，方向还成立吗？ | ❌ 仍不成立——方案本身与锚点矛盾 | ✅ 成立——方向本就对，只是论证有瑕疵 |
| **病灶在哪** | 在"用什么衡量成功"（标尺错了） | 在"怎么论证的"（链条断了/前提假） |
| **对症动作** | 重选标尺，可能整个方案推翻重来 | 修链条/换前提，方案主体不动 |

**替换测试是闸门：** 拿不准是方向还是质量时，问"**假设我把论证修到滴水不漏，这个方案的方向还对吗？**"
- 对 → 结构层问题，回生成层
- 不对 → 价值层问题，回意图层

**红旗：** 把什么都说成价值层（全盘重来）= 偷懒怕精细；把什么都说成结构层（只改产出）= 怕退回起点的麻烦。两者都是不分流的逃逸。
