# 发散探索（Divergent Exploration）

> **定位：** deep-intent-collaboration 阶段 1.5「发散探索」的完整方法论展开。
> 意图澄清（阶段一）完成后、进入方案选择前，需要系统化地"带着问题去探险"时由主技能引用。是阶段一（收敛）与阶段二（执行）之间的插入阶段。

## 顶层使命

**发散不是漫无目的，而是"带着问题去探险"——在收敛前先把空间看清楚。**

用户做决策前，往往需要了解已有的最佳实践、可选的技术方案/设计思路、以及不同选择的代价和影响。本模块提供系统化的发散机制，确保用户做选择前"看见了选项空间"。

反查元依据：**"用户做选择前，看见选项空间了吗？"**

---

## 何时进入此阶段

不是所有任务都需要发散探索，只有以下情况才进入：
1. **技术选型未定**（用户未明确说"必须用X"）
2. **设计方案存在多种可行路径**（AI看到多个合理选项）
3. **用户在阶段一表现出不确定**（"我不确定哪种方式更好"、"有没有其他做法"）
4. **需要了解行业最佳实践/已有经验**（首次接触某领域）

**不进入：** 用户已明确技术选型（"必须用React"）；任务明确只有一条可行路径；低风险可逆操作（修错字、加注释）；用户明确说"直接做，不用查"。

---

## 发散探索的三层结构

| 层级 | 目标 | 动作 | 过载防御 |
|------|------|------|----------|
| **情境层** | 确认"在这个上下文里值得探索什么" | 基于锚点定义3-5个探索维度 | 不超过5个，每个1句话 |
| **信息层** | 收集"在这个维度上有哪些选项" | 搜索每个维度，收集2-3个典型方案 | 每个维度只收集"代表性"方案，不穷举 |
| **对比层** | 呈现"不同选项间如何权衡" | 6维评估 + 差异分析 + 影响分析，拉回对话框自顶向下 | 先呈现对比矩阵，再按需深入细节 |

---

## 情境层：定义探索维度

**目标：基于锚点，确定值得探索的方向。**

动作：基于阶段一的锚点（目的、约束、成功标准、本质需求），提炼出 3-5 个探索维度；每个维度用 1 句话描述，聚焦"这个维度上有什么不同的选择"；向用户确认这些维度是否准确。

**过载防御：** 不超过 5 个维度（信息过载）；每个维度 1 句话（不展开）；维度必须基于锚点（不能无目的发散）。

---

## 信息层：双向搜索（内部优先 + 外部补充）

**目标：在每个维度上，收集"值得考虑的"选项，不是穷举所有选项。**

**搜索方向：先内部，再外部。** 不跳过内部直接搜外部，避免"搜了一堆外部方案，结果项目内早有答案"的浪费。

### 搜索触发判断

对每个探索维度，AI 先自评确信度，决定搜索深度：

```
当前维度 → AI 自评：我的知识足够判断吗？
    ├── ✅ 足够（确信度 🟢）
    │   → 直接呈现已知选项，标注「🤖 基于AI知识」
    ├── ⚠️ 部分足够（确信度 🟡）
    │   → ① 先内部搜索 → 够了吗？
    │       ├── ✅ → 呈现，标注「📖 基于项目内部资料」
    │       └── ❌ → ② 外部搜索 → 呈现，标注「🌐 基于外部调研」
    └── ❌ 不足（确信度 🔴）
        → ① 先内部搜索 → ② 不够再外部搜索 → 呈现，标注来源
```

### 内部搜索（优先）

**搜索对象（按优先级）：**
1. **项目代码/配置文件** — 项目已经用了什么？有什么基础设施？
2. **`.experience-log/`** — 过往经验是否有涉及类似决策的复盘？
3. **`wiki/`** — 团队是否有已有的知识沉淀？
4. **当前会话上下文** — 前面的对话中是否提到过相关信息？

动作：对每个维度先搜内部资料；收集已有信息中的代表性选项（2-3个即可）；每个选项 1 句话描述核心特点，标注来源。

### 外部搜索（补充）

**仅当内部搜索后仍信息不足时执行。** 搜索对象：行业最佳实践/主流方案对比；竞品/同行做法；官方文档/技术规范；社区讨论/已知坑点。

动作：对每个维度搜外部资料（网页、文档、社区）；只收集"代表性"方案（每个维度 2-3 个）；每个方案 1 句话描述核心特点，标注来源链接。

### 状态快照（轻量版，对抗凭感觉搜）

双向搜索不是"搜一锤子买卖"——每轮结束产出**简化版状态快照**，下一轮基于快照而非凭印象继续：

```
[本轮搜索结束]
- 已确认：[2-3 条，每条标来源类型 🤖📖🌐]
- 待解：[1-2 条，按对决策的影响排序]
- 下一轮要搜的 sub-query：[列表，每条标注基于本轮哪个发现]
```

把"凭感觉搜"变成"有依据地细化"，对抗长上下文里早期发现被遗忘。**完整版 Plan-Execute-Reflect 循环 + reflection 充分性终止 + 强制 plan review** 见 `external-solution-research.md`。

### 深度调研模块（升级路径）

**当外部搜索需要系统化深度时，升级到 `external-solution-research.md`（REQUIRED SUB-SKILL）。**

**升级触发条件（满足任一）：** 探索性技术问题 / 普通外部搜索（搜 1-2 次就停）不足以覆盖方案空间 / 方案的可靠度与已知坑点对用户决策关键。

该模块按 Deep Research 原理设计，提供：**Plan-Execute-Reflect 显式循环**（Phase 0 问题分解 → Phase 1 广度 → 【强制】Plan Review → Phase 2 深度 → Phase 3 reflection 充分性判断 → Phase 4 整合）+ **reflection 充分性终止**（自问三题）+ **强制 plan review**（👤 用户决策的分叉点）+ **不确定性三态标注** + **深度≠准确护栏 + tool-call hacking 反查** + 可靠度四级分级、坑点三层防御（含反方证据）、产出硬性模板。

> **核心使命对齐：** 该模块为兑现"坑和不确定性可预见 ≠ 零坑"——把可能带坑的方案无标签推给用户 = 核心罪过。其产出格式（可靠度分级 + 坑点预警 + 不确定性三态 + 反方证据）为强制要求，不可降级。

### 过载防御（双向搜索共用）

- 每个维度只收集 2-3 个代表性选项（不穷举）
- 每个选项 1 句话（不展开）
- 不报搜索过程细节（"我查了5个网站"），只报搜索结果和来源
- 内部搜索有答案后不再外部搜索（避免信息过载）
- 搜索结论向用户透明化：标注每个选项的来源类型

## 对比层：系统化评估框架

**目标：让用户看清不同选项间的权衡，而不是被淹没在细节里。** 用 6 维系统化评估框架，评估后按自顶向下分层披露。

### 6 维评估维度（必选框架）

| 维度 | 问题 | 评估方式 |
|------|------|----------|
| **① 意图对齐度** | 这个方案是否满足本质需求锚点？ | 对齐 ✅ / 有张力 ⚠️ / 偏离 ❌ |
| **② 约束兼容性** | 是否在三个边界（范围/深度/技术）内？ | 兼容 ✅ / 部分兼容 ⚠️ / 冲突 ❌ |
| **③ 实施代价** | 需要多少时间/人力/学习成本？ | 低 / 中 / 高（附估算理由） |
| **④ 潜在风险** | 什么会出错？失败影响面多大？ | 列出 1-2 个关键风险，标注确信度 |
| **⑤ 长期影响** | 选了这个，对后续有什么锁定效应？ | 切换成本 / 生态绑定 / 技术债 |
| **⑥ 内部契合度** | 项目已有的代码/经验/工具链是否支持？ | 高契合 ✅ / 需改造 ⚠️ / 不兼容 ❌ |

**6 维评估是必选框架，不是可选框架。** 认为某维度"没意义"本身就是一个评估结论，必须说明为什么没意义，不能跳过。

### 差异分析（必做）

对比矩阵只展示"各方案在各维度的得分"，不展示"方案间的核心区别"。**差异分析**是显式提取方案间的关键分叉点：

> **差异分析：**
> - A 与 B 的核心区别在于「长期影响」：A 无锁定但实施代价中，B 有生态绑定但上手快
> - 如果本质需求是「低承诺快速试错」，A 更对齐；如果是「快速出活不在乎后续」，B 更合适

**差异分析必须显式、结构化地提取。** 隐含在行文中的差异用户需要自己提炼——那是甩给用户的审核义务。

### 影响分析（必做）

**影响分析**不是"如果选错了会怎样"的风险分析，而是**方案选择本身带来的结构性影响**：

| 影响类型 | 问题 |
|----------|------|
| **切换成本** | 未来如果想换方案，需要付出什么？ |
| **生态绑定** | 选了之后，会不会被某个技术/供应商锁定？ |
| **学习曲线** | 团队需要学新东西吗？学习期间的生产力损失？ |
| **连带依赖** | 这个方案会引入哪些额外的依赖/工具/流程？ |

### 反方证据（涉及比较性判断时必做）

**Deep Research 反 one-sided 护栏：** DeepTRACE 实证调研系统在 debate 类查询上"高度 one-sided"——只呈现一方观点。

> **触发：** 当某结论涉及**比较性/推荐性判断**（"X 比 Y 好""X 是最佳""推荐 X"）时，**强制列反方观点**。
> 搜到反方 → 列反方论据 + 来源；搜不到 → 标「未发现公开反方讨论」（不能默认"无争议"）。

与差异分析**互补**：差异分析是"中立对比"，反方证据是"主动找反对意见"。完整版（含反方查询模板）见 `external-solution-research.md` 坑点侦察第二层。

### 不确定性诚实标注（简化版）

**搜不到 ≠ 没问题，也 ≠ 不能答。** 笼统标"未知"会让用户无从决策，按三态之一标注（完整定义见 `external-solution-research.md`）：

| 状态 | 标注 | 处理 |
|------|------|------|
| 能力边界型（工具够不到） | 「超出能力边界」 | 标注 + 建议接什么工具 |
| 信息矛盾型（来源冲突） | 「存在争议」 | 列各方 + 让用户判 |
| 信息不足型（搜得到的不够） | 「证据不足」 | 还需搜什么 + 当前最强假设 |

**铁律：** 绝不用"安全""无坑""成熟稳定""无争议"描述"搜不到"。

### 自顶向下披露

评估完成后，按三层结构逐层呈现给用户：

1. **第一层：6 维对比矩阵**（骨架）→ 先看整体
2. **第二层：差异分析 + 影响分析**（仅当骨架通过）→ 深入关键分叉
3. **第三层：单个方案详细评估**（仅当用户要求）→ 深入某个方案

**披露铁律：** 必须拉回对话框（不能只甩文件路径）；自顶向下逐层披露（不能全文倾倒）；每层都要对齐锚点（不能脱离锚点对比）。

---

## 阶段 1.5 验收

**用户能回答："我看到了不同选项间的权衡，现在我能做出选择。"** 答得出 = 发散探索达成目标。答不出 = 信息过载或对比不清晰，需重新调整。

## 与主技能的协作关系

| 主技能环节 | 本模块对接点 |
|-----------|-------------|
| 阶段一·本质需求锚点 | 本模块的探索维度、方案排序均对照该锚点 |
| 阶段一·规则2.5（类比只是入口） | 类比帮用户理解后，方案选择必须回到本模块的 6 维评估（类比是入口，6 维评估才是判据） |
| 审核框架·方案审核 | 本模块的 6 维评估 + 差异分析 + 影响分析是方案审核的输入素材（见 `audit-framework.md`） |
| 决策透明化框架 | 每个方案标注推理依据🤖📖🌐✅ + 确信度🟢🟡🔴（见 `decision-transparency.md`） |

**与其他 reference 的协作：** 外部搜索的深度强化 → `external-solution-research.md`；方案审核的校验手段 → `audit-framework.md`。

**独立使用时：** 若未被主技能引用，本模块需用户先明确"要探索的问题 + 成功标准"，再进入三层结构——相当于自带一个迷你版的意图锚定。
