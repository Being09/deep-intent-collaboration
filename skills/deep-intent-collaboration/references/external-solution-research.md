# 外部方案深度调研模块（External Solution Research）

> **定位：** 本模块是 deep-intent-collaboration 阶段 1.5「信息层·外部搜索」的深度强化路径。
> 当外部搜索需要系统化深度（探索性技术问题、需可靠度分级与坑点预警）时，由主技能引用加载。
> 可独立使用，也可被其他技能的"外部搜索"环节引用复用。

## 顶层使命

**系统化地先广后深调研外部解决方案，产出必须自带可靠度分级 + 已知坑点预警。**

核心目的不是消灭坑（前沿实验性方案必然有未知风险），而是**让坑变得可预见**——
前沿实验性方案可接受，但**不可 unlabeled 地把带坑方案当成熟方案推给用户**。

unlabeled = 核心罪过。把可能带坑的方案无标签推给用户，比"搜得不够广"更不能接受。

每个动作都能反查元依据：**"这个产出，坑是否可预见了？"**

---

## 何时启用本模块

**升级触发条件（满足任一即从普通外部搜索升级到本模块）：**

- 问题属探索性技术问题（无确定解，需从业界/学术界找方案）
- 普通外部搜索（搜 1-2 次就停）不足以覆盖方案空间
- 方案的可靠度与已知坑点对用户决策关键

**不启用：**

- 事实查询（"Python GIL 是什么"）→ 直接答
- 用户已明确技术选型（"必须用 X"）→ 走主技能快速对齐
- 低风险可逆操作 → 直接做
- 用户明确说"直接做，不用查"

> **与主技能阶段 1.5 的关系：** 本模块是阶段 1.5「外部搜索（补充）」的深度强化版。
> 阶段 1.5 的外部搜索是"每个维度收集 2-3 个代表性方案，不穷举"；
> 本模块用于当这个深度不够、需要系统化铺开 + 深挖 + 坑点侦察时。

---

## 核心机制：迭代式调研流程

**核心命题：调研不是"搜一次就收"，是"带着意图锚定，先广后深，迭代收敛"。**

### 流程总览

```
┌─ 第一轮：广度搜索（Breadth Pass）
│   目标：铺开方案空间，不深入任何一个
│   产出：方案候选清单（每方案一句话 + 初步来源标签）
│
├─ 意图锚定评估 + 排序
│   对照主技能阶段一的锚点（本质需求/约束/成功标准）
│   每个候选方案评：意图对齐度 + 实施性价比
│   取 Top N（默认 N=3）进入深度搜索
│
├─ 第二轮：并行深度搜索（Depth Pass）
│   优先分派子 Agent 并行深挖 Top N
│   客户端不支持子 Agent → 回退单 Agent 按优先级逐个深挖
│   每个方案深挖：技术细节 / 实现参考 / 复杂度 / 依赖
│
├─ 分叉判断：遇到不确定的关键问题？
│   ├─ 是 → 回到广度搜索（新一轮迭代，聚焦不确定点）
│   └─ 否 → 进入坑点侦察 + 收敛产出
│
└─ 输出：带可靠度分级 + 坑点预警的调研报告
```

### 第一轮：广度搜索（Breadth Pass）

**目标：回答"这个领域有哪些主要技术路线/解决方案"，铺开空间。**

**动作：**
1. 用你可用的网页搜索能力，搜索该问题的技术路线概览（不深入任何一个）
2. 每个候选方案一句话描述核心特点 + 初步来源标签
3. 不过早评判哪个"最好"——这一轮只负责"看见空间"

**过载防御：**
- 不在这一轮深挖任何单方案（深挖留给第二轮）
- 候选方案控制在 5-8 个（过多 = 信息过载，过少 = 空间没铺开）

### 意图锚定评估 + 排序

**目标：用主技能阶段一的锚点过滤方案，选出最值得深挖的 Top N。**

**动作：**
1. 对照锚点（本质需求/约束/成功标准），对每个候选方案评两个维度：
   - **意图对齐度**：这个方案满足本质需求吗？在约束边界内吗？
   - **实施性价比**：实施代价（时间/人力/学习成本）vs 预期收益
2. 排序，取 Top N（默认 N=3，可由用户调整）
3. 排序理由向用户透明化（不只给排序结果，要给"为什么这个排前"）

**铁律：排序必须对照锚点，不能凭"我觉得这个更主流/更先进"。**
"主流/先进"是表面理由，要追到"为什么对锚点来说更对齐"。

### 第二轮：并行深度搜索（Depth Pass）

**目标：对 Top N 方案做深度挖掘。**

**并行子 Agent 机制（优先）：**
- 若客户端支持分派子 Agent，**每个方案分配一个子 Agent 独立深挖**
- 子 Agent 的任务：深挖单方案的技术细节、实现参考、复杂度、依赖
- 各子 Agent 独立返回结果后，由主 Agent 汇总

**回退策略（客户端不支持子 Agent 时）：**
- 单 Agent 按排序优先级逐个深挖（逻辑等价，但串行）

**每个方案深挖的内容（子 Agent 或单 Agent 都要覆盖）：**
- 技术核心要点（这个方案的本质机制是什么）
- 实现参考（有无可复用的代码/库/教程，来源在哪）
- 复杂度评估（实现难度、性能开销）
- 依赖清单（需要引入什么额外的东西）

**工具抽象原则：** 描述搜索能力时用自然语言（"使用你可用的网页搜索/网页抓取能力"），
不写死具体工具名——保证跨 Agent 客户端可移植。

### 分叉判断 + 迭代控制

**何时触发新一轮迭代（回到广度搜索）：**

- 深度搜索中冒出新的不确定关键问题（例如：发现方案 A 依赖某技术，但该技术的可行性又不确定）
- 该不确定问题**会实质影响方案排序或选择**

**最大迭代次数：默认 3 轮（含首轮）。**

- 达到 3 轮仍有不确定 → 停止迭代，把"剩余不确定点"作为调研报告的一部分如实呈现给用户
- **不无限搜索**——过载防御铁律。3 轮搜不清的，要么是问题本身太前沿（标注"前沿未定"），要么是信息源够不到（标注"超出能力边界"）

**迭代轮次的透明化：** 每轮迭代都要向用户说明"这是第几轮，为什么还要再搜一轮"。

## 可靠度分级（四级标签）

**目标：每个方案必须有可靠度标签，让用户知道"这个方案有多可信"。**

| 等级 | 标签 | 判定标准 |
|------|------|----------|
| **L1** | 🟢 工业验证 | 已有成熟商业产品在使用 / 高 Star 开源项目（Star 数由 Agent 结合领域判断，不同领域基准不同） |
| **L2** | 🔵 学术可信 | 高引用学术论文 / 顶会发表（如 SIGGRAPH、ACM TOG、High Performance Graphics） |
| **L3** | 🟡 学术参考 | 低引用论文 / 预印本 / 未被广泛复现的研究 |
| **L4** | 🟠 玩具 Demo | 个人 demo / 单文件实现 / 无生产环境验证 |

**操作化说明：**
- Star 阈值、高引/低引阈值**由 Agent 结合具体领域判断**（图形学领域的高 Star 标准和 Web 前端不同，不给硬性数字）
- 标签必须基于**可追溯的证据**（看到哪个产品在用/哪个仓库多少 Star/哪篇论文多少引用），不能凭感觉
- 边界情况（如"中等 Star 但被知名项目依赖"）→ 取较低等级 + 备注，宁可保守

## 坑点侦察（三层防御）⭐ 核心模块

**这是本模块的核心——unlabeled 推带坑方案 = 核心罪过。**
坑的信息完整性有**两个方向都要防**：
- **漏报**（有坑没说）= 用户踩坑
- **误报**（没坑瞎编）= 破坏用户对调研结论的信任

三层防御覆盖这两个方向。

### 第一层：反向搜坑（主动挖坑，防漏报）

**目标：对每个进入深度搜索的方案，主动搜索其已知问题，而不是等用户踩到。**

**动作：**
对每个方案，组织反向查询，搜索方向包括：
- 该方案/库名的 issue、bug、problem、limitation、artifact
- 专门的坑点信息源（按可靠度）：
  - 记录"什么方法会 FAIL"的死路数据库
  - 从开发者社区提取真实抱怨的聚合站点
  - 官方 issue tracker / 代码仓库的 Issues
  - 社区论坛中精确匹配痛点短语的讨论

**搜索方向由 Agent 根据方案性质组织，不写死固定查询串**——
不同领域、不同类型的方案，坑点聚集的地方不同。

### 第二层：坑点真伪验证（防误报）⭐

**目标：搜到的坑必须验证真伪，防止 Agent 为显得"调研充分"编造不存在的坑。**

**背景：** Agent 可能在压力下（想显得调研完整、想凑够坑点数量）hallucinate 不存在的坑。
LLM 编造"已知 issue"是已知模式，必须防。

**动作：**
每个声称的坑，要求给出**可追溯来源**：
- issue 链接 / 论坛帖子链接 / 文档章节 / 代码行
- 查无来源的坑 → 一律标注「⚠️ 未验证，疑似编造」，**绝不当作事实陈述**

**对接主技能的「事实查证三层」**（可追溯来源 / 可信度 / 可证方法）：
- 可追溯来源？→ 给出具体链接/出处
- 可信度？→ 评估来源权威性、时效性
- 可证方法？→ 给不出处时，说清怎么验证

**硬规则：查无可靠来源、又说不清怎么验证的坑，一律当未证实，不采信，不写入调研报告。**

### 第三层：未知诚实标注（防漏报的最重要防线）

**目标：搜不到坑 ≠ 没有坑。必须明确标注风险等级，绝不默认"安全"。**

三种状态，必须显式标注其一：

| 状态 | 含义 | 标注方式 |
|------|------|----------|
| **已验证坑** | 搜到且有可靠来源的坑 | 标注具体坑 + 来源链接 |
| **疑似坑** | 搜到但来源不可靠/未验证 | 标注「疑似，待验证」+ 现有线索 |
| **未知风险** | 搜不到任何公开讨论的坑点 | 标注「⚠️ 未发现公开讨论的坑点，风险等级：未知」 |

**铁律：绝不用"安全""无坑""成熟稳定"这类词描述"搜不到坑"的情况。**
"搜不到"只能产出"未知风险"标注，不能产出"安全"结论。

> **这一层是兑现顶层使命的关键**——
> "未知"是可预见的状态（明确告诉用户"我不确定有没有坑"），
> unlabeled 是假装没坑。两者天差地别。
> 用户可以接受"风险未知"的方案（自己评估是否值得冒险），
> 但不能接受被 unlabeled 地推进坑里。

## 信息源策略

### 通用优先级（适用所有领域）

官方文档/规范 > 高 Star 开源仓库 > 顶会论文 > 技术权威博客 > 社区讨论

### 游戏开发特有信息源（按可靠度）

**学术顶会（最高可靠度，但常需付费/视频）：**
- SIGGRAPH Technical Papers / ACM Transactions on Graphics (TOG)
- High Performance Graphics
- Ke-Sen Huang 会议论文索引（图形学论文聚合）

**工业顶会（工业实践 + 学术结合）：**
- GDC Vault（Game Developers Conference）
- Advances in Real-Time Rendering in 3D Graphics and Games 课程

**社区权威（实践经验，免费可访问）：**
- realtimerendering.com 门户
- Alan Zucconi 等知名技术博客
- 引擎官方文档（Unity / Unreal / Godot）

**开源实现（可读代码）：**
- GitHub 高 Star 仓库
- Shadertoy（着色器 demo 聚合）

**社区讨论（真实使用痛点，适合坑点侦察）：**
- Reddit r/GraphicsProgramming / r/gamedev
- gamedev.stackexchange

### 可扩展

本模块以游戏开发为主要场景，但信息源策略可泛化——
遇到其他领域的探索性问题，Agent 应根据领域特征组织对应的信息源分层。

## 能力边界与升级接口

**核心命题：诚实标注工具能力边界，比假装全能更有价值。**

### 当前文本/网页搜索能力可够到

- 论文页面、技术博客、官方文档、社区讨论
- 开源仓库（README、Issues、代码）
- 公开可抓取的网页内容

### 够不到的（明确标注，不假装）

- **学术论文 PDF 全文**（付费墙）→ 只能搜到摘要/页面
- **会议视频内容**（GDC 演讲等）→ 无法观看/转写
- **需登录/交互的动态页面**（某些 demo、付费内容）

### 升级接口（预留触发条件，未来接新工具即生效）

当调研触发以下需求时，标注"超出当前能力，建议接扩展工具"，并预留接口位：

| 触发条件 | 建议的扩展工具类型 | 接口位状态 |
|----------|-------------------|-----------|
| 需要学术 PDF 全文 | 学术搜索 API / 论文获取工具 | 预留 |
| 需要会议视频内容 | 视频转写 / browser-use 类浏览器自动化 | 预留 |
| 需要动态页面交互 | browser-use / Playwright 类浏览器自动化 | 预留 |
| 需要付费数据库内容 | 对应数据库访问工具 | 预留 |

**遇到能力边界时的动作：**
1. 明确告知用户"该资料超出当前工具能力"
2. 说明"接 ___ 类工具后可覆盖"
3. 不因此停止调研——用现有能力尽量覆盖，把够不到的部分如实标注

## 产出格式（硬性模板，拦截 unlabeled）

**调研报告必须包含以下结构，缺一不可。这是验收项。**

### 报告模板

````markdown
## 调研报告：[问题]

### 方案概览（按意图锚定排序）

| 排序 | 方案 | 可靠度 | 意图对齐 | 实施代价 | 坑点状态 |
|------|------|--------|----------|----------|----------|
| 1    | A    | 🟢 L1  | 对齐 ✅   | 中       | ⚠️ 1 已验证坑 |
| 2    | B    | 🟡 L3  | 有张力 ⚠️ | 低       | ❓ 未知风险 |
| 3    | C    | 🟠 L4  | 对齐 ✅   | 低       | ⚠️ 疑似坑待验证 |

### 各方案详情

#### 方案 A：[名称]
- **技术核心：** [本质机制一句话]
- **可靠度依据：** [哪个产品在用 / 仓库 Star / 论文引用，给可追溯证据]
- **实现参考：** [代码/库/教程来源]
- **复杂度：** [实现难度 + 性能开销]
- **依赖：** [需要引入什么]
- **已知坑点：**
  - [坑 1]（来源：[链接]）✅ 已验证
  - [坑 2]（来源：[线索]）⚠️ 待验证

（B、C 同结构）

### 综合建议

对照锚点 [一句话复述本质需求]，推荐 [方案 X]，理由是 [对照锚点的理由]。
需注意：[可靠度状态] + [坑点状态]，[是否需用户进一步判断]。

### 能力边界说明（若有）

本次调研中，[某某资料] 超出当前工具能力（[原因]），建议 [如何补全]。
````

### 论证结构可视化（ASCII 树状图）——硬性产出要求

**综合建议及每个关键判断，必须以 ASCII 树状图形式，完整详尽地递归展示"论点-论据"的论证结构。**

调研报告的推荐不是"我觉得方案 A 好"的断言，而是一棵从论点到叶节点证据的论证树。不展示 = 论证黑箱 = 用户无法检验推理链条，与主技能「决策透明化」和「审核框架·论证质检五步」直接冲突。

**树状图四要求：**

1. **递归到底**——每个论据若仍含子判断（非单一可验证事实），继续展开到叶节点（可独立验证的单一事实 / 证据 / 来源）。
2. **完整详尽**——支持论点的所有论据都列出；**不挑拣有利的、隐藏不利的**——漏列反方论据 = 论证造假。
3. **标注来源**——叶节点标可靠度（🟢🔵🟡🟠）+ 信息溯源（🔗📄📓💬），符号体系见主技能「决策透明化」。
4. **含反论据**——对立论据（他方案优势、本方案弱点）纳入树中，显式标注处理方式（被抵消 / 部分成立 / 待用户权衡）。

**示例（综合建议的论证树）：**

```
推荐方案 A（论点）
├─ 论据1：意图对齐度最高
│   ├─ 子论据：满足本质需求「[锚点一句话]」🟢💬
│   └─ 子论据：在约束「[具体约束]」边界内 🟢💬
├─ 论据2：可靠度 L1（工业验证）🟢🔗
│   └─ 证据：[产品 X 在用] / [仓库 N 万 Star]
├─ 论据3：坑点可预见
│   ├─ 已验证坑：[坑1]（来源：链接）✅
│   └─ 未知风险：已显式标注 ⚠️（不默认"安全"）
└─ 反论据处理
    ├─ 方案B更轻量 → 但[具体理由]不抵消A的对齐优势
    └─ 方案C更新 → 未知风险更高，已纳入权衡
```

> 本要求是主技能「审核框架·论证质检五步」（论点明确 / 论据相关 / 论据充分 / 论据可信 / 反驳处理）在调研产出上的可视化落地——树状图把质检从"检查清单"变成"可见结构"，用户扫一眼就能看出哪根论据断了。

### 验收铁律

- 任何方案若**缺可靠度标签** → 视为 unlabeled → **禁止交付**
- 任何方案若**缺坑点状态**（哪怕是"未知风险"）→ 视为 unlabeled → **禁止交付**
- 综合建议若**未对照锚点** → 视为未对齐 → **禁止交付**
- 综合建议若**缺 ASCII 论证树**（论点-论据递归结构未展开）→ 视为论证黑箱 → **禁止交付**
- 能力边界若**被隐瞒**（假装够到了实际没有）→ 视为欺骗 → **禁止交付**
