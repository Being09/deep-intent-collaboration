# 外部方案深度调研模块（External Solution Research）

> **定位：** deep-intent-collaboration 阶段 1.5「信息层·外部搜索」的深度强化路径。
> 外部搜索需要系统化深度（探索性技术问题、需可靠度分级与坑点预警）时由主技能引用。可独立使用。
>
> **本模块按 Deep Research 原理设计**——显式 Plan-Execute-Reflect 循环（参考 Gemini Deep Research、LangChain open_deep_research）+ 人在环 plan review + reflection 充分性终止 + 不确定性三态标注。原理限于纯 prompt 可落地范围；端到端 RL / 向量库 / 真 context isolation 等工程化能力不在本模块范围。

## 顶层使命

**系统化地先广后深调研外部解决方案，产出必须自带可靠度分级 + 已知坑点预警 + 不确定性诚实标注。**

核心目的不是消灭坑（前沿实验性方案必然有未知风险），而是**让坑和不确定性变得可预见**——前沿实验性方案可接受，但**不可 unlabeled 地把带坑方案当成熟方案推给用户**。**unlabeled = 核心罪过**，比"搜得不够广"更不能接受。

反查元依据：**"这个产出，坑和不确定性是否可预见了？"**

---

## 何时启用

**升级触发条件（满足任一即从普通外部搜索升级到本模块）：**
- 探索性技术问题（无确定解，需从业界/学术界找方案）
- 普通外部搜索（搜 1-2 次就停）不足以覆盖方案空间
- 方案的可靠度与已知坑点对用户决策关键

**不启用（边界要严，避免简单任务被重流程拖累）：** 事实查询（"Python GIL 是什么"）→ 直接答；用户已明确选型（"必须用 X"）→ 主技能快速对齐；低风险可逆操作 → 直接做；用户明确说"直接做不用查"。

> **与主技能阶段 1.5 的关系：** 本模块是阶段 1.5「外部搜索（补充）」的深度强化版。阶段 1.5 外部搜索是"每个维度 2-3 个代表性方案，不穷举"；本模块用于这个深度不够、需系统化铺开+深挖+坑点侦察时。

## 核心机制：Plan-Execute-Reflect 显式循环

**核心命题：调研不是"搜一次就收"，是"带着意图锚定，先广后深，每轮基于已有发现重新规划，直到能充分回答原始需求"。**（对齐 Gemini Deep Research plan-search-synthesize + open_deep_research per-round reflection，本模块用显式 5 阶段循环：）

```
┌─ Phase 0：意图锚定 + 研究问题分解（query decomposition）
│   对照主技能阶段一锚点，把用户问题拆成可独立回答的 sub-questions
│
├─ Phase 1：广度搜索（Breadth Pass）
│   铺开方案空间，不深入任何一个；产出候选清单（每方案一句话 + 初步来源标签）
│
├─ 【强制】Plan Review（人在环）
│   把研究计划展示给用户改：Top N 方案 + 每个深挖什么 + 预期 sub-query
│   ↳ 这是分叉点，👤 用户决策（主技能阶段二分叉点校验）
│
├─ Phase 2：深度搜索（Depth Pass）
│   优先分派子 Agent 并行深挖 Top N；客户端不支持 → 回退单 Agent 串行
│   每方案深挖：技术核心 / 实现参考 / 复杂度 / 依赖
│
├─ Phase 3：Reflection 充分性判断（自问三题）
│   主终止条件——充分性驱动，不是轮次预算
│   不充分 → 产出下一轮 sub-query 清单，回 Phase 2
│
└─ Phase 4：整合产出
    带可靠度 + 坑点 + 反方证据 + 溯源链的调研报告
```

### Phase 0：意图锚定 + 研究问题分解

对照主技能阶段一锚点（本质需求/约束/成功标准），把用户原始问题拆成 3-7 个可独立回答的 sub-questions，每个标注"解决原始需求的哪个部分"。

**铁律：分解必须对照锚点，不能凭"我觉得该查这些"。** "主流/常见"是表面理由，要追到"为什么对锚点需要这个答案"。

### Phase 1：广度搜索（Breadth Pass）

用网页搜索能力搜该问题的技术路线概览（不深入任何一个）。每个候选一句话 + 初步来源标签，不过早评判"最好"——这轮只"看见空间"。候选控制在 5-8 个（多=过载，少=空间没铺开）。

产出第一份状态快照 → 进入 Plan Review。

### 【强制】Plan Review（人在环）

**研究计划必须能被用户 review 和修改——分叉点，不是 AI 自主决策。**（对齐 Gemini Deep Research "review and iterate on the plan" + open_deep_research "human-in-the-loop planning"，与主技能"决策权在人"哲学一致）

**广度搜索产 Top N 方案后、深度搜索前，强制停下展示研究计划给用户：**

```
基于广度搜索，我整理出研究计划（Top N=3）：

| 排序 | 方案 | 可靠度初判 | 意图对齐 | 我打算深挖什么 |
|------|------|-----------|----------|---------------|
| 1    | A    | 🟢 L1?    | 对齐     | 实现参考 + 已知坑 + 长期维护 |
| 2    | B    | 🟡 L3?    | 有张力   | 边界条件 + 迁移成本 |
| 3    | C    | 🟠 L4?    | 对齐     | 是否有生产级案例 |

预期 sub-query：A:[q1/q2] / B:[q1] / C:[q1]
预计需要 1-2 轮深度搜索。
```

**等用户回应才进 Phase 2。用户可改：** 缩范围（"跳过 B"） / 加深度（"C 多挖一层"） / 扩范围（"加个 D"） / 改优先级（"B 才是首选"） / 提前终止（"直接基于现有信息推荐"）。

> 对接主技能阶段二分叉点校验——是分叉点，**必须 👤 用户决策**，AI 不自主推进到 Phase 2。用户授权"你定吧"按授权陷阱处理（停下让他看清代价）。

### Phase 2：深度搜索（Depth Pass）

**子 Agent 优先：** 客户端支持分派子 Agent 时，**每个方案分一个子 Agent 独立深挖**——真 context isolation（每子 Agent 独立上下文，避免长链路腐烂）。各子 Agent 返回后主 Agent 汇总。客户端不支持 → 回退单 Agent 串行按优先级深挖（逻辑等价，无 isolation，状态快照格外重要）。

**每方案深挖覆盖：** 技术核心 / 实现参考（代码/库/教程 + 来源）/ 复杂度（难度 + 性能开销）/ 依赖清单。

**工具抽象：** 描述搜索能力用自然语言（"使用你可用的网页搜索/抓取能力"），不写死工具名——保跨客户端可移植。

每轮结束产出状态快照 → 进 Phase 3。

### Phase 3：Reflection 充分性判断（自问三题）

**核心命题：终止判断是"能否充分回答原始需求"，不是"第几轮"——充分性驱动，不是轮次预算驱动。**（对齐 open_deep_research per-section reflection）

**每轮深度搜索后必自问三题（顺序不可跳）：**

```
1. 我现在能回答用户的【原始本质需求】了吗？
   ├─ 能 → 进入 Phase 4
   └─ 不能 → 继续问下一题

2. 还差什么【关键不确定点】？（列出，不凭感觉）
   ├─ 列得出 → 继续问下一题
   └─ 列不出（都覆盖了）→ 实际上能答，回第 1 题

3. 再搜一轮【能否实质改变排序/选择】？
   ├─ 能 → 产出下一轮 sub-query 清单，回 Phase 2
   └─ 不能 → 进入 Phase 4（边际收益，停）
```

**3 轮上限（反过载护栏，非主终止条件）：** 达 3 轮仍不确定 → 停止迭代，剩余不确定按三态标注如实呈现（不无限搜索）。3 轮搜不清 → 问题太前沿（"前沿未定"）或信息源够不到（"超出能力边界"）。

**透明化 + sub-query 纪律：** 每轮向用户说明"第几轮、reflection 判断、为什么还要/不再搜"。下一轮每条 sub-query 必须标注"为什么搜这个（基于上轮哪个具体发现）"——把"凭感觉搜"变成"有依据地细化"，对抗 DEFT 的 Lack of Analytical Depth。

### Phase 4：整合产出

把全部搜索发现整合成带可靠度 + 坑点 + 反方证据 + 溯源链的调研报告。产出格式见后文「产出格式」。

## 可靠度分级（四级标签）

每个方案必须有可靠度标签——让用户知道"这方案有多可信"。

| 等级 | 标签 | 判定标准 |
|------|------|----------|
| **L1** | 🟢 工业验证 | 已有成熟商业产品使用 / 高 Star 开源项目（Star 数由 Agent 结合领域判断） |
| **L2** | 🔵 学术可信 | 高引用学术论文 / 顶会发表（如 SIGGRAPH、ACM TOG、HPG） |
| **L3** | 🟡 学术参考 | 低引用论文 / 预印本 / 未被广泛复现的研究 |
| **L4** | 🟠 玩具 Demo | 个人 demo / 单文件实现 / 无生产环境验证 |

**操作化：** Star/引用阈值**由 Agent 结合领域判断**（不同领域基准不同，不给硬性数字）；标签必须基于**可追溯证据**（哪个产品在用/多少 Star/多少引用），不能凭感觉；边界情况（"中等 Star 但被知名项目依赖"）→ 取较低等级 + 备注，宁可保守。

## 坑点侦察（三层防御）⭐ 核心模块

**核心——unlabeled 推带坑方案 = 核心罪过。** 坑信息完整性两个方向都要防：**漏报**（有坑没说=用户踩坑）+ **误报**（没坑瞎编=破坏信任）。三层防御覆盖两个方向，第二层兼任"反方证据"（防比较性判断偏颇）。

### 第一层：反向搜坑（主动挖坑，防漏报）

对每个进入深度搜索的方案，主动搜其已知问题（不等用户踩到）。反向查询方向：方案/库名的 issue、bug、problem、limitation、artifact；专门坑点信息源（按可靠度）：记录"什么方法会 FAIL"的死路数据库、聚合开发者真实抱怨的社区、官方 issue tracker / 仓库 Issues、社区论坛精确匹配痛点短语的讨论。

**搜索方向由 Agent 根据方案性质组织，不写死固定查询串**——不同领域、不同类型方案，坑点聚集地不同。

### 第二层：坑点真伪验证 + 反方证据（防误报 + 防 one-sided）⭐

**双重职责——**

| 职责 | 防什么 | 动作 |
|------|--------|------|
| **① 坑点真伪验证** | 防误报（LLM 在"想显得完整"压力下编造"已知 issue"） | 每个声称的坑要求**可追溯来源**（issue 链接 / 论坛帖子 / 文档章节 / 代码行）。查无来源 → 标「⚠️ 未验证，疑似编造」，**绝不作事实陈述** |
| **② 反方证据** | 防 one-sided（DeepTRACE 实证：调研系统在 debate 类查询上高度 one-sided） | **触发：结论涉及比较性/推荐性判断**（"X 比 Y 好""X 是最佳""推荐 X"）时，强制搜反方（"X 的批评""X 失败案例""X vs Y 反对意见"）。搜到→列反方论据+来源；搜不到→标「未发现公开反方讨论」（不能默认"无争议"） |

两职责**互补不可省**：①针对**技术坑点**（方案有什么 bug），②针对**比较性判断的偏颇**（为什么不该选它）。

**对接主技能「事实查证三层」**（可追溯来源 / 可信度 / 可证方法）：给具体链接/出处 → 评估权威性时效性 → 给不出处时说清怎么验证。**硬规则：查无可靠来源、又说不清怎么验证的坑，一律当未证实，不采信不写入。**

### 第三层：不确定性三态标注（防漏报的最重要防线）

**搜不到 ≠ 没问题，也 ≠ 不能答。** "不知道"必须结构化，不能笼统标"未知"。参考 DEFT 失败模式（arXiv:2512.01948）头号失败——"agents 宁可编造也不说不知道"。单一"未知"标签让用户无从决策；拆成三态，各自对应不同处理动作：

| 状态 | 含义 | 标注 | 处理动作 |
|------|------|------|----------|
| **能力边界型** | 工具够不到（付费墙/视频内容/动态页面/需登录） | 「超出能力边界」 | 标注 + 建议接什么工具可补（学术搜索 API / 视频转写 / browser-use 类） |
| **信息矛盾型** | 多方来源冲突，无法判断谁对 | 「存在争议」 | 列各方观点 + 各自可靠度 + 让用户判（不替用户判是非） |
| **信息不足型** | 搜得到的就是不够，无法下结论 | 「证据不足」 | 标注还需搜什么 + 当前最强假设（明确标"假设非结论"） |

**铁律：绝不用"安全""无坑""成熟稳定""无争议"描述"搜不到"。** "搜不到"只能产出上述三态之一，不能产出"安全"结论。用户可接受"风险未知/存在争议/证据不足"（自己评估是否冒险），但不能接受被 unlabeled 推进坑里或推向错误结论。

## 状态快照机制（对抗长上下文腐烂）

**长链路调研的头号敌人是上下文腐烂——搜得越多，早期关键发现越易被淹没。** 参考 Anthropic 多 agent 用 isolated context 对抗腐烂（~15× token 代价）；纯 prompt 做不到真 isolation，但**用结构化中间产物可模拟**。

**每轮搜索（Phase 1 广度 / Phase 2 每轮深度）结束产出固定格式状态快照：**

```
[第 N 轮结束状态]
- 已确认事实：[列表，每条带可靠度 L1-L4 + 来源]
- 已排除方案/假设：[列表 + 排除理由]
- 待解问题：[列表，按对决策的影响排序]
- 下一轮 sub-query：[列表，每条标注"基于上轮哪个发现"]
```

**纪律：下一轮搜索基于快照而非完整历史继续**——强迫每轮"提炼"而非"堆积"，对抗早期信息被淹没。

**局限性（不假装等效）：** 这是 prompt 版近似，效果不如工程化 subagent context isolation。若客户端支持子 Agent，Phase 2 的并行深挖已是真 isolation；状态快照主要服务"主 Agent 跨轮汇总"和"不支持子 Agent 时的串行回退"。

## 深度≠准确护栏

**反直觉但实证支持：搜索深度增加时，事实准确率反而下降。** arXiv:2605.06635 发现 increased search depth **consistently degrades** factual accuracy while surface-level citation metrics appear reassuring（引用越像样，事实越不准）。

**硬规则：**

> **每条结论必须独立绑定到具体来源 + 可靠度等级**，不因"搜了很多轮""引用很全""经过了 reflection"放松溯源。
>
> 搜得多 ≠ 查得严。深度搜索后反而要更严地反查每条结论的溯源——"引用丰富感"最易让模型和用户放松警惕。

**tool-call hacking 反查：** 每条结论必须能指出"它来自哪轮的哪次搜索/哪个来源"。对不上的 → 标「⚠️ 未溯源，疑似后填的引用」。堵住"搜了一堆但结论是脑补的，引用是事后装饰"的失败（LinkedIn: tool-call hacking；DeepHalluBench: unfaithful to source）。

## 信息源策略

**通用优先级（适用所有领域）：** 官方文档/规范 > 高 Star 开源仓库 > 顶会论文 > 技术权威博客 > 社区讨论。

### 游戏开发特有信息源（附录，按可靠度）

> 本模块以游戏开发为主要场景，但策略可泛化——遇其他领域探索性问题，Agent 应根据领域特征组织对应信息源分层。

- **学术顶会（最高可靠度，常需付费/视频）：** SIGGRAPH Technical Papers / ACM TOG / High Performance Graphics / Ke-Sen Huang 论文索引
- **工业顶会（工业实践+学术）：** GDC Vault / Advances in Real-Time Rendering 课程
- **社区权威（实践经验，免费）：** realtimerendering.com / Alan Zucconi 等技术博客 / 引擎官方文档（Unity/Unreal/Godot）
- **开源实现（可读代码）：** GitHub 高 Star 仓库 / Shadertoy
- **社区讨论（真实痛点，适合坑点侦察）：** Reddit r/GraphicsProgramming / r/gamedev / gamedev.stackexchange

## 能力边界

**诚实标注工具能力边界，比假装全能更有价值。**

**当前文本/网页搜索能力可够：** 论文页面、技术博客、官方文档、社区讨论、开源仓库（README、Issues、代码）、公开可抓取的网页内容。

**够不到的（明确标注，不假装）——归入"能力边界型不知道"：** 学术论文 PDF 全文（付费墙）→ 只能搜摘要/页面；会议视频内容（GDC 演讲等）→ 无法观看/转写；需登录/交互的动态页面（某些 demo、付费内容）。

**遇到能力边界时：** 明确告知"该资料超出当前工具能力"（标能力边界型不知道）+ 说明"接 ___ 类工具后可覆盖"（学术搜索 API / 视频转写 / browser-use 类 / 对应数据库访问工具）。不因此停止调研——用现有能力尽量覆盖，把够不到的如实标注。

## 产出格式（硬性模板，拦截 unlabeled）

**调研报告必须含以下结构，缺一不可（验收项）：**

````markdown
## 调研报告：[问题]

### 研究问题分解（Phase 0 产出）

原始需求：[一句话锚点]
拆成的 sub-questions：
1. [sub-q1]——解决[原始需求的哪个部分]
2. ...

### 方案概览（按意图锚定排序）

| 排序 | 方案 | 可靠度 | 意图对齐 | 实施代价 | 坑点状态 | 反方证据状态 |
|------|------|--------|----------|----------|----------|------------|
````

### 论证结构可视化（结构化信息输出，调研产出场景）——硬性产出要求

**综合建议及每个关键判断，必须以结构化信息输出形式（默认 ASCII 树状图），完整详尽递归展示"论点-论据"的论证结构。**

调研推荐不是"我觉得方案 A 好"的断言，而是从论点到叶节点证据的论证树。不展示 = 论证黑箱 = 用户无法检验推理链，与主技能「决策透明化」+「审核框架·论证质检五步」直接冲突。

> **本节是 `output-typography.md`「结构化信息输出」横切切面在【调研产出时】场景的落地。**
> 共性规则（形式谱系 / 先图后文 / 三共性要求）的正式定义见 `output-typography.md`。本节只保留**调研场景独有**的细节。

**调研场景独有标注：** 叶节点标可靠度（🟢🔵🟡🟠）+ 信息溯源（🔗📄📓💬），符号体系见 `decision-transparency.md`——每个证据带可靠度分级和来源溯源。

**示例（综合建议论证树骨架）：**
```
推荐方案 A（论点）
├─ 论据1：意图对齐度最高 🟢💬
│   ├─ 满足本质需求「[锚点一句话]」
│   └─ 在约束「[具体约束]」边界内
├─ 论据2：可靠度 L1（工业验证）🟢🔗 → [产品 X 在用]/[仓库 N 万 Star]
├─ 论据3：坑点可预见 → 已验证坑 + 能力边界型不确定性已标
├─ 论据4：反方证据已处理 → [反方论据]→[为何不抵消优势] 或「未发现公开反方」
└─ 反论据处理 → B 更轻量/C 更新 为何不抵消 A 的对齐优势
```

### 验收铁律（增强版）

缺任一项 → **禁止交付**：
- 方案**缺可靠度标签** → unlabeled
- 方案**缺坑点状态**（哪怕是三态之一）→ unlabeled
- 方案**缺反方证据状态**（涉及比较性判断时）→ one-sided
- 综合建议**未对照锚点** → 未对齐
- 综合建议**缺结构化论证树** → 论证黑箱
- 结论**无法映射回具体搜索/来源**（tool-call hacking 反查不过）→ 标「⚠️ 未溯源」或回补
- 能力边界**被隐瞒**（假装够到实际没有）→ 欺骗
- **深度≠准确护栏失守**：经多轮搜索的结论，溯源要求反而比单轮松
