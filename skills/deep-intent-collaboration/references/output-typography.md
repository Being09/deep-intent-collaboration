# 输出排版原则（Output Typography）

> **定位：** 本模块是 deep-intent-collaboration 「输出排版原则」横切段的方法论展开。
> 当 AI 在实际输出时需要决定如何排版（引用块/视觉层级/符号使用）时由主技能引用加载。
> 横切贯穿全程——与各阶段"做什么"无关，只约束"输出怎么排版"。

## 顶层使命

**排版是协作的一部分——关键信息视觉弱化，等于信息丢失。**

分析、判断、需要用户决策的点，如果被埋进长段落或弱视觉标记里，用户扫一眼抓不到 → 协作降级成"用户得自己从字堆里捞重点"。
排版不是装饰，是把"哪些是重点"这件事在视觉层兑现。

每个动作都能反查元依据：**"这个关键信息，视觉权重配得上它的重要性吗？"**

---

## 引用块（`>`）的合法用途（只这三类）

引用块是**次要标记**（视觉权重最低），只用于：

| 合法用途 | 含义 | 示例 |
|----------|------|------|
| **① 引用户原话** | 复述用户刚说的话，便于对照 | `> 用户原话："[内容]"` |
| **② 次要注释/旁白** | 不影响主流程的补充说明、meta 标注 | "本节是 X 的特例应用" |
| **③ 文档内示例脚本** | 本文档里的话术模板（演示"说什么"，非"怎么排版"） | 见各阶段"话术模板" |

**禁止：** 分析、判断、推荐、决策请求、核心命题、铁律**一律不进引用块**。把它们塞进 `>` = 用最弱的视觉标记承载最重要的信息 = 信息丢失。

> **文档示范 ≠ 输出照搬：** 本文档里的话术模板大量用引用块，那是"示例脚本"的文档约定（演示**说什么**），不是"实际输出就该这么排版"。实际输出时，照搬引用块外壳 = 视觉降级。模板教你的是内容结构，排版按本原则重新决定。

---

## 关键信息的视觉层级

按信息重要性，从弱到强用对标记：

**规则 1：需用户决策的点 → 独立成块 + 小标题 + 粗体收尾。**
决策点是最该突出的。用 `###` 小标题单列（如 `### 🧭 分叉点：是否删旧表`），选项列清单，最后用**粗体**给出明确的决策请求（如 **`👉 你的决定？`**）。不混在长段落里，不藏在引用块末尾。

```text
### 🧭 分叉点：是否删旧表

- **A** 保留：零风险，留旧数据
- **B** 删除：干净，但不可逆

我倾向 B（理由：旧数据已迁移）。
**👉 你的决定？**
```

**规则 2：关键分析/判断/推荐 → 粗体领句。**
重要判断不要溶解在平铺文字里。用粗体把"这句话是结论"标出来，让用户扫一眼就抓到核心。

**规则 3：长内容 → 结构化（小标题/列表/表格），不用长段落淹没。**
超过几行的连续散文，按逻辑拆成小标题 + 列表/表格。用户对结构的扫描速度远高于对长段落。

**规则 4：信息密度匹配视觉权重。**
- 越重要的信息 → 越强的视觉标记（小标题 > 粗体 > 普通正文 > 引用块）
- 次要的补充 → 弱标记（引用块、括号）
- **错配即失效**：重要信息用弱标记（引用块/正文末尾）= 用户抓不到；次要信息用强标记（到处加粗）= 真正的加粗失去提示力（"狼来了"）
- **"欠标记"是最隐蔽的违规**：把关键结论用普通正文陈述（没加粗、没小标题）不像"塞进引用块"那样有可见的违规形状，最容易被自己忽略，却同样让用户抓不到重点。规则2要求的"关键判断粗体领句"是一条主动义务——**该突出而没突出，等同于突出失败**，不是"中性默认"。自检时别只盯"是不是多加了不该加的标记"，也要盯"该加的是不是漏了"。

## 符号叙事：强调类 emoji 各归其位（防串味）

强调类符号承载情绪基调，**一符一义**——同一符号既当"你的舞台"又当"小心陷阱"，会把"请你定夺"误传成"这里有问题"。约定如下：

| 角色 | 符号 | 语义 | 情绪基调 |
|------|------|------|----------|
| 分叉点 / 用户决策时刻 | 🧭 | 岔路口，方向归你定 | 积极、强调（你的高光时刻） |
| 真陷阱（授权陷阱等） | 🪤 | 字面陷阱，需警惕 | 警告 |
| 评估"部分/有张力" | ⚠️ | 中性 caution，待判 | 提醒（非定罪） |
| 裁决通过 / 不通过 | ✅ / ❌ | 结论 | 确定 |

**反模式**：用 ⚠️ 标决策点 = 把用户行使主权的高光时刻套上警示符，在情绪层暗示"这里有问题/你选错了"。决策点用 🧭，真警告才用 🪤 / ⚠️，别混。（确信度 🟢🟡🔴、推理依据 🤖📖🌐✅、信息溯源 🔗📄📓💬、人机分工 👤🤖🤝 已各自成族，不在本约定范围。）

## 结构化信息输出（横切切面，与逻辑校验平级）

> **本节是「结构化信息输出」的 canonical 定义。**
> 它收口了原本散落在三处的"可视化"要求——`audit-framework.md`「逻辑校验可视化」（管**校验时**）、`external-solution-research.md`「论证结构可视化」（管**调研产出时**）、`learning-design-flow.md`「贯穿要求·论证结构可视化」（管**学习设计产出时**）。三处的共性规则统一收口于此，各场景保留自己的场景特定细节（示例图、符号体系、验收铁律）。

### 核心命题

**含实质内容需传达的输出（分析/建议/解释/论证/方案/结论/诊断/推荐），天然带有"结构"，不是线性文字。**

实质内容的核心都是**关系**——前提与结论、锚点与选择、论点与论据、因果与互斥、要素与从属。关系天然是结构：谁支撑谁、谁反驳谁、哪根线断了。用线性文字叙述 = 把结构压扁，断链 / 隐藏前提 / 矛盾线全都藏进句子里，校验者和读者都看不出哪根线断了。

**结构化信息输出**把关系从线性文字还原成可见结构——是一类和"逻辑校验"平级的横切切面：逻辑校验负责"判断关系成不成立"，结构化信息输出负责"把关系以可检视的形态呈现出来"。前者做完了后者没做 = 校验做了等于没做（关系依然隐在线性文字里）。

每个动作都能反查元依据：**"这次的输出含实质内容吗？含的话，我以可见结构呈现了它的关系吗？还是压成了线性文字？"**

### 排版铁律·先图后文（结构化图示不可替代文本）

**适用范围（触发线：含实质内容即触发，取宽口径）：** 凡**含实质内容需传达的输出**——分析、建议、解释、论证、方案、结论、诊断、推荐——一律**先输出结构化图示，再输出正文**。实质内容几乎都带关系结构（分析含因果、建议含理由、解释含要素关系），触发面取宽口径，**疑则触发**。

**豁免（仅这三类，不可扩大）：**
- **纯确认/回应**（"好的"、"收到"、"明白"）——无实质内容
- **纯动作/状态**（"我先改这个文件"、"执行中"）——动作本身无关系结构
- **纯事实直答**（"Python GIL 是 X"、"这个函数返回 Y"）——单一可验证事实，无关系需呈现

**防漏洞铁律：** AI 不得用"这次输出不算逻辑相关"逃避先图后文——触发线已从"逻辑相关"扩大到"含实质内容"，"这只是个建议/解释/说明"恰恰是实质内容的伪装，不是豁免理由。豁免仅限上述三类（确认/动作/事实直答），**实质内容类一律不豁免**。

**结构化图示是"关系的骨架"，正常文本内容是"血肉"——缺一不可。**

输出顺序固定为：
1. **先输出结构化图示**——表达本次输出的逻辑关系（论证树、校验关系图、收敛树）或概念关系（要素关系、流程）。
2. **再输出正常文本内容**——论证展开、细节说明、话术、建议。

为什么先图后文：图让用户扫一眼看出"整体结构 / 哪根线断了 / 哪里有矛盾"，文本随后承载无法压进图的细节。若只有图无文，细节丢失；若只有文无图，结构压扁——**两者都不可少，且图在前**。

**反模式：** 用图替代文本（"图都画了，正文不用再解释了"）= 丢失细节；或把图塞在文末当"补充" = 用户读完线性文字才看到结构，结构失去了"先给全局"的作用。

### 形式谱系（密度 × 可读性优先级，含形态边界）

结构化信息输出有四种形式，默认按**信息密度 × 可读性**综合排序从高到低选择，但每种形式有**形态边界硬约束**——不是单纯"越高优先级越好"，而是"看被表达信息的结构形态定形式"。

| 优先级 | 形式 | 强项 | 形态边界（何时用 / 何时降级） |
|--------|------|------|------------------------------|
| **1（默认）** | **ASCII 图（树状优先）** | 信息密度最高；中文（等宽）排版整齐稳定；层级结构一目了然 | 层级 / 从属 / 分叉关系 → **首选树状图**；互斥 / 网状 / 多对多关系 → 用节点 + 连线关系图（仍属 ASCII 图） |
| **2** | **Mermaid 图** | 时序图 / 流程图渲染清晰，适合状态流转、时序先后 | **胜在时序与流程**；但**元素过多时渲染后图形稀疏、不便扫读 → 此时降回 ASCII**（密度优先） |
| **3** | **SVG 图** | 复杂精确图形（几何关系、空间布局） | 仅当 ASCII / Mermaid **无法精确表达**时用；大部分语义 / 概念关系场景 ASCII 可替代，不必上 SVG |
| **4** | **Markdown 表格 / 层级缩进列表** | 简单枚举 / 对照 / 二维矩阵 | 结构简单到"不需要图"时用；**能图不表**——关系结构明显时压成表格 = 降级 |

**选择决策（先看形态，再定形式）：**

```
被表达信息的结构形态是？
│
├─ 层级 / 从属 / 分叉（前提→结论、论点→论据、锚点→选择）
│     → ASCII 树状图（优先级 1）
│
├─ 互斥 / 网状 / 多对多（约束互斥、循环论证、矛盾线）
│     → ASCII 关系图·节点+连线（优先级 1，标关系类型）
│
├─ 时序 / 流程 / 状态流转
│     ├─ 元素少 → Mermaid（优先级 2，渲染清晰）
│     └─ 元素多 → 降回 ASCII（优先级 1，密度优先）
│
├─ 复杂精确图形（几何 / 空间布局，ASCII 画不准）
│     → SVG（优先级 3）
│
└─ 简单枚举 / 对照（无关系结构）
      → 表格 / 列表（优先级 4）
```

### 三共性要求（所有形式通用）

无论选哪种形式，都必须满足：

1. **递归到底 + 完整详尽**——论据若仍含子判断（非单一可验证事实），继续展开到叶节点（可独立验证的单一事实 / 证据 / 来源）。支持论点的所有论据全列出；**不挑拣有利的、隐藏不利的**——漏列反方论据 = 论证造假。
2. **标注来源与校验状态**——节点 / 连线上标校验结果（`✅ 成立` / `❌ 断裂` / `⚠️ 待验` / `🤖 AI 可判` / `👤 用户定`）；叶节点标可靠度（🟢🔵🟡🟠）+ 信息溯源（🔗📄📓💬，符号体系见 `decision-transparency.md`）。校验结果落在结构上，不是另起一段文字复述。
3. **含反论据 / 矛盾线**——对立论据（他方案优势、本方案弱点）、矛盾线（互斥的约束对）纳入结构中，显式标注处理方式（被抵消 / 部分成立 / 待用户权衡 / ❌ 不自洽）。矛盾线不画出来 = 假装自洽。

## 与主技能的协作关系

本模块在以下环节与 deep-intent-collaboration 主技能对接：

| 主技能环节 | 本模块对接点 |
|-----------|-------------|
| 全程输出 | 引用块禁忌、视觉层级四规则、符号叙事——约束 AI 每一次实际输出的排版 |
| 阶段二·产出审核披露铁律 | 铁律一"拉回对话框"受本模块规则1（决策点独立成块）支撑 |
| 审核框架 / 调研产出 / 学习设计产出 | 「结构化信息输出」横切切面——校验时、调研产出时、学习设计产出时的可视化要求统一收口于本模块 |
| 防 AI 自我溃败·红旗清单 | "塞进引用块/欠标记"红旗由本模块定义判定标准 |

**独立使用时：** 若未被主技能引用，本模块可独立作为"AI 输出排版规范"使用——核心命题（视觉权重匹配信息重要性）不依赖协作上下文。
