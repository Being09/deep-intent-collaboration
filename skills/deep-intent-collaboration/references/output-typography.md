# 输出排版原则（Output Typography）

> **定位：** deep-intent-collaboration「输出排版原则」横切的方法论展开。
> AI 实际输出时需要决定排版（引用块/视觉层级/符号使用）时由主技能引用。横切贯穿全程——只约束"输出怎么排版"。

## 顶层使命

**排版是协作的一部分——关键信息视觉弱化，等于信息丢失。**

分析、判断、需要用户决策的点，如果被埋进长段落或弱视觉标记里，用户扫一眼抓不到 → 协作降级成"用户得自己从字堆里捞重点"。排版不是装饰，是把"哪些是重点"在视觉层兑现。

反查元依据：**"这个关键信息，视觉权重配得上它的重要性吗？"**

---

## 引用块（`>`）的合法用途（只这三类）

引用块是**次要标记**（视觉权重最低），只用于：

| 合法用途 | 含义 | 示例 |
|----------|------|------|
| **① 引用户原话** | 复述用户刚说的话便于对照 | `> 用户原话："[内容]"` |
| **② 次要注释/旁白** | 不影响主流程的补充说明、meta 标注 | "本节是 X 的特例应用" |
| **③ 文档内示例脚本** | 本文档里的话术模板（演示"说什么"，非"怎么排版"） | 见各阶段"话术模板" |

**禁止：** 分析、判断、推荐、决策请求、核心命题、铁律**一律不进引用块**。塞进 `>` = 用最弱的视觉标记承载最重要的信息 = 信息丢失。

> **文档示范 ≠ 输出照搬：** 本文档话术模板大量用引用块，那是"示例脚本"的文档约定（演示**说什么**），不是"实际输出就该这么排版"。照搬引用块外壳 = 视觉降级。模板教你的是内容结构，排版按本原则重新决定。

---

## 关键信息的视觉层级（四规则）

按信息重要性，从弱到强用对标记：

**规则 1：需用户决策的点 → 独立成块 + 小标题 + 粗体收尾。**
用 `###` 小标题单列（如 `### 🧭 分叉点：是否删旧表`），选项列清单，最后**粗体**给明确决策请求（如 **`👉 你的决定？`**）。不混长段落，不藏引用块末尾。

**规则 2：关键分析/判断/推荐 → 粗体领句。**
重要判断不溶解在平铺文字里。用粗体把"这句话是结论"标出来。

**规则 3：长内容 → 结构化（小标题/列表/表格），不用长段落淹没。**

**规则 4：信息密度匹配视觉权重。**
- 越重要 → 越强标记（小标题 > 粗体 > 普通正文 > 引用块）
- 次要补充 → 弱标记（引用块、括号）
- **错配即失效**：重要信息用弱标记 = 抓不到；次要信息用强标记（到处加粗）= 真正的加粗失去提示力
- **"欠标记"是最隐蔽的违规**：关键结论用普通正文陈述（没加粗、没小标题）不像"塞进引用块"那样有可见违规形状，最易被自己忽略，却同样让用户抓不到。**该突出而没突出 = 突出失败**，不是"中性默认"。

## 符号叙事：强调类 emoji 各归其位（防串味）

强调类符号承载情绪基调，**一符一义**——同一符号既当"你的舞台"又当"小心陷阱"，会把"请你定夺"误传成"这里有问题"。

| 角色 | 符号 | 语义 | 情绪基调 |
|------|------|------|----------|
| 分叉点 / 用户决策时刻 | 🧭 | 岔路口，方向归你定 | 积极、强调（你的高光时刻） |
| 真陷阱（授权陷阱等） | 🪤 | 字面陷阱，需警惕 | 警告 |
| 评估"部分/有张力" | ⚠️ | 中性 caution，待判 | 提醒（非定罪） |
| 裁决通过 / 不通过 | ✅ / ❌ | 结论 | 确定 |

**反模式：** 用 ⚠️ 标决策点 = 把行使主权的高光时刻套警示符，情绪层暗示"这里有问题/你选错了"。决策点用 🧭，真警告才用 🪤 / ⚠️。

（确信度 🟢🟡🔴、推理依据 🤖📖🌐✅、信息溯源 🔗📄📓💬、人机分工 👤🤖🤝 已各自成族，不在本约定范围。）

## 结构化信息输出（横切切面，与逻辑校验平级）

> **本节是「结构化信息输出」的正式定义。**
> 收口了原本散落三处的"可视化"要求——`audit-framework.md`（管**校验时**）、`external-solution-research.md`（管**调研产出时**）、`learning-design.md`（管**学习设计产出时**）。三处的共性规则统一收口于此，各场景保留自己的场景特定细节。

### 核心命题

**含实质内容需传达的输出（分析/建议/解释/论证/方案/结论/诊断/推荐），天然带有"结构"，不是线性文字。**

实质内容的核心都是**关系**——前提与结论、锚点与选择、论点与论据、因果与互斥、要素与从属。关系天然是结构。用线性文字叙述 = 把结构压扁，断链 / 隐藏前提 / 矛盾线全藏进句子里，校验者和读者都看不出哪根线断了。

**结构化信息输出**把关系从线性文字还原成可见结构——与"逻辑校验"平级的横切切面：逻辑校验负责"判断关系成不成立"，结构化信息输出负责"把关系以可检视形态呈现"。

反查元依据：**"这次的输出含实质内容吗？含的话，我以可见结构呈现了它的关系吗？还是压成了线性文字？"**

### 排版铁律·先图后文

**适用范围（触发线：含实质内容即触发，取宽口径）：** 凡**含实质内容需传达的输出**——分析、建议、解释、论证、方案、结论、诊断、推荐——一律**先输出结构化图示，再输出正文**。实质内容几乎都带关系结构，**疑则触发**。

**豁免（仅这三类，不可扩大）：**
- **纯确认/回应**（"好的""收到""明白"）——无实质内容
- **纯动作/状态**（"我先改这个文件""执行中"）——动作本身无关系结构
- **纯事实直答**（"Python GIL 是 X""这个函数返回 Y"）——单一可验证事实

**防漏洞铁律：** AI 不得用"这次输出不算逻辑相关"逃避先图后文——触发线已从"逻辑相关"扩大到"含实质内容"，"这只是个建议/解释/说明"恰恰是实质内容的伪装，不是豁免理由。

**结构化图示是"关系的骨架"，正常文本内容是"血肉"——缺一不可。** 输出顺序固定：
1. **先输出结构化图示**——表达逻辑关系（论证树、校验关系图、收敛树）或概念关系（要素关系、流程）
2. **再输出正常文本内容**——论证展开、细节说明、话术、建议

**反模式：** 用图替代文本（"图都画了，正文不用再解释"）= 丢失细节；把图塞文末当"补充" = 用户读完线性文字才看到结构。

### 形式谱系（密度 × 可读性，含形态边界）

四种形式，按**信息密度 × 可读性**综合排序，但每种有**形态边界硬约束**——看被表达信息的结构形态定形式。

| 优先级 | 形式 | 强项 | 形态边界 |
|--------|------|------|----------|
| **1（默认）** | **ASCII 图（树状优先）** | 信息密度最高；中文（等宽）排版整齐稳定；层级一目了然 | 层级/从属/分叉→**首选树状图**；互斥/网状/多对多→用节点+连线关系图 |
| **2** | **Mermaid 图** | 时序/流程图渲染清晰 | **胜在时序流程**；但**元素过多时渲染稀疏、不便扫读 → 降回 ASCII** |
| **3** | **SVG 图** | 复杂精确图形（几何关系、空间布局） | 仅当 ASCII/Mermaid **无法精确表达**时用 |
| **4** | **Markdown 表格 / 层级缩进列表** | 简单枚举/对照/二维矩阵 | 结构简单到"不需要图"时用；**能图不表** |

**选择决策：**
```
被表达信息的结构形态是？
├─ 层级/从属/分叉（前提→结论、论点→论据、锚点→选择）→ ASCII 树状图
├─ 互斥/网状/多对多（约束互斥、循环论证、矛盾线）→ ASCII 关系图·节点+连线
├─ 时序/流程/状态流转
│   ├─ 元素少 → Mermaid
│   └─ 元素多 → 降回 ASCII（密度优先）
├─ 复杂精确图形（几何/空间布局，ASCII 画不准）→ SVG
└─ 简单枚举/对照（无关系结构）→ 表格/列表
```

### 节点与区域的框定方式（ASCII 图子规则）

**核心命题：框定手段按"对齐脆弱性"递增排序——能轻则轻。**

ASCII 图里"框住一块内容"有四种手段，对齐义务从轻到重：

| 框定方式 | 形态 | 对齐义务 | 适用场景 |
|----------|------|----------|----------|
| **无框**（连线即框定） | 节点=文字本身，靠连线表达关系 | 无 | **关系图节点**（前提-结论、锚点-选择、要素-从属） |
| **单侧开放角标** | 仅左上 `┌─`、左下 `└─`，右侧开放 | 仅左侧对齐 | **分组/分区/嵌套层级** |
| **四角十字** | 四角各放 `+`，不连线 | 四角大致对位（容错高） | **独立容器/对象语义**（架构图组件、UML 实体） |
| **闭合线框** | `┌┐└┘` + 连续竖横线全周闭合 | 全周连续闭合（最脆弱） | **避免使用**——仅在必须手绘几何精确图形时 |

**为什么这条成立：** 树状/叉状连线"只要连到内容行就整齐"，因为只有**单侧**有标记，右侧开放——没有"需要对齐的东西"。闭合线框易歪，因为它要求**左右竖线 + 上下横线 + 四角同时精确闭合**，任一处算错一格全图失准。框定手段的选择，本质是在选"接受多少对齐义务"——义务越少，图越稳。

**闭合线框在 CJK 混排下结构性必歪（不是"小心点就行"）：** 等宽终端里，半角字符（`a` `(` `─`）占 1 显示列，全角字符（汉字、全角 `（）`、全角 `""`）占 2 显示列。闭合框要求右竖线 `│` 落在固定列，但框内只要混入中文，每行内容的**显示宽** = `Σ(半角×1 + 全角×2)` 两两不同，右竖线必然散落到不同列——且字节长度、字符数、显示宽三者各不相等，手工按任一口径计算都算不准。**结论：框内含中文时，闭合线框的右竖线无法对齐，这是显示机制的硬约束，不是排版失误。**

> **硬规则：框内含中文（任何全角字符）时，不得使用闭合线框——改用单侧开放角标或无框。** 纯半角内容（全英文/数字/符号）三者口径合一，方可闭合。

**例外（本原则不适用）：**
- **表格**：`|` 竖线**本身就是对齐结构**（列对齐是语义的一部分），不是"框定"
- **Markdown 原生代码块**（```` ``` ````）：语法自带围栏
- **需要手绘几何精确图形**（如真正 UML 类图全要素、空间布局示意图）：ASCII 表达几何本就受限，此时闭合线框是手绘几何的必要手段

本节是「形式谱系」的子规则——形式谱系决定**整体用哪种形式**，本节决定**选定 ASCII 后，节点和区域的"边框"用什么形态**。

### 三共性要求（所有形式通用）

无论选哪种形式，都必须满足：

1. **递归到底 + 完整详尽**——论据若仍含子判断（非单一可验证事实），继续展开到叶节点（可独立验证的单一事实/证据/来源）。支持论点的所有论据全列出；**不挑拣有利的、隐藏不利的**——漏列反方论据 = 论证造假。
2. **标注来源与校验状态**——节点/连线上标校验结果（`✅ 成立`/`❌ 断裂`/`⚠️ 待验`/`🤖 AI 可判`/`👤 用户定`）；叶节点标可靠度（🟢🔵🟡🟠）+ 信息溯源（🔗📄📓💬，符号体系见 `decision-transparency.md`）。校验结果落在结构上，不是另起一段文字复述。
3. **含反论据 / 矛盾线**——对立论据（他方案优势、本方案弱点）、矛盾线（互斥的约束对）纳入结构，显式标注处理方式（被抵消/部分成立/待用户权衡/❌ 不自洽）。矛盾线不画出来 = 假装自洽。
